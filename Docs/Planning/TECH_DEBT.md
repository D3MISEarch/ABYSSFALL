# Tech Debt

Non-blocking items only — nothing here blocks a merge on its own. Each entry is tracked, not implemented, until it's picked up as its own scoped task with its own ADR if it touches architecture. Source: Stage 5 independent review (see [`Docs/Roadmap/STAGE_5_LOOT_AFFIXES.md`](../Roadmap/STAGE_5_LOOT_AFFIXES.md), "Independent review" section) and [ADR-018](../ADR/018_PROCEDURAL_ITEM_GENERATION.md) deferred decisions.

| Item | Severity | Likely milestone | Notes |
|---|---|---|---|
| One authoritative identity minter is required for future multiplayer | High (architectural precondition) | Multiplayer foundation stage (pre-Phase 6 co-op work in `PROJECT_OVERVIEW.md`) | `ItemIdentityService` currently assumes one authoritative `RuntimeSession` per build. Two concurrent sessions mutating the same build will collide. Needs network-authority design before online co-op begins — see [ADR-018](../ADR/018_PROCEDURAL_ITEM_GENERATION.md) "Deferred decisions." |
| Malformed item identity sequence warnings may be added | Low | Opportunistic / Stage 6 hardening | `ItemIdentityService.observe()`/`configure()` currently accept a mismatched or malformed snapshot silently by falling back to sequence 1 rather than logging a warning. Not currently causing failures, but a warning would surface a corrupted save earlier. |
| Flattened affix modifiers should be reconsidered before tooltip UI and crafting | Medium | Before tooltip UI or crafting work begins | `ItemInstance.affixes` currently flattens rolled affixes into the existing stat-modifier array (kept compatible with equipment stat application). Tooltip UI and crafting/reroll systems will likely need structured affix groups instead of flattened records — see [ADR-018](../ADR/018_PROCEDURAL_ITEM_GENERATION.md) "Affix representation." |
| Affix eligibility pools may eventually require indexing | Low | When catalog scale or profiling justifies it | `AffixCatalog.eligible_definitions()` is a linear scan today. Per the Engineering Constitution's "optimize only after measuring" law, this stays as-is until profiling shows it matters. |
| Rarity rules may eventually move from hardcoded data to definitions | Medium | Before map/elite modifiers need dynamic rarity budgets | `LootRarity` currently hardcodes rarity tier rules (min/max affixes, prefix/suffix caps). This is a direct instance of the "data-driven over hardcoded" law not yet being fully met for rarity specifically — acceptable for now, but should become catalog-driven data before elite/map modifiers need to adjust budgets dynamically. |
| Normalize the pre-existing lowercase `docs/` and uppercase `Docs/` directory split | Medium | Before the documentation tree grows significantly; own dedicated migration PR | Found during the PR #36 documentation audit — see [`Docs/README.md`](../README.md) "Known inconsistency" section. **Owner:** Codex / repository engineering. **Scope:** mechanical path migration (moving files) and updating every relative link that currently points at `docs/...`, not a documentation-content rewrite. **Risk:** Git tracks `docs/` and `Docs/` as distinct case-sensitive paths; a case-insensitive Windows checkout silently merges them into one folder, while case-sensitive tooling (e.g. this repository's own Ubuntu-runner CI) would see two separate top-level directories, so the ambiguity is currently masked rather than resolved. **Timing/constraint:** perform as its own dedicated migration PR — do not bundle this into gameplay work or into an unrelated documentation change, since it touches link targets referenced from `.github/workflows/`, `AGENTS.md`, and multiple `Docs/` files. |
